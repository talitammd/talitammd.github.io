JavaScript 执行机制

JavaScript 的执行机制主要涉及到其单线程特性、事件循环以及任务队列等概念。下面详细解释这些概念以及它们如何共同影响 JavaScript 的执行。

单线程

JavaScript 是单线程的，这意味着它一次只能执行一个任务。这确保了代码执行的顺序性和一致性，但同时也带来了一个问题：当遇到需要等待的操作（如 I/O 操作、定时器、网络请求等）时，整个线程会被阻塞，直到该操作完成。

异步编程

为了解决这个问题，JavaScript 引入了异步编程的概念。异步编程允许 JavaScript 在等待某个操作完成的同时执行其他任务。但如果同步代码依赖的是异步任务执行完的结果，怎么办呢？

给出的解决方法就是回调函数，当你调用一个异步API并传递一个回调函数时，一旦异步操作完成，事件循环会收到通知，并自动将相应的回调函数推入任务队列。

常见的异步任务有：

  1. 定时器：setTimeout 和 setInterval
  2. 建立网络连接：fetch API 、 XMLHttpReques、 http、https 或其他第三方库（如 axios、request）
  3. 读取网络流数据
  4. 向文件写入数据：fs
  5. Ajax提交
  6. 请求数据库服务
  7. 动态import()

异步操作在Node.js中的执行环境：

  1. 事件循环：与浏览器环境不同的是，Node.js的事件循环包括更多的阶段（如timers、pending callbacks、idle, prepare、poll、check、close callbacks等），这些阶段用于处理不同类型的异步任务。
  2. 多线程：虽然JavaScript在Node.js中仍然是单线程执行的，但Node.js内部使用了多线程来处理某些类型的异步任务。这些线程由Node.js的底层库管理，当Node.js代码执行一个异步I/O操作时，它会将任务委托给libuv库，然后立即返回并继续执行后续的代码。libuv库会根据操作系统的功能（例如，Linux的epoll、Windows的IOCP等）来高效地处理这些异步I/O任务。当异步I/O操作完成时，libuv库会通知Node.js，然后Node.js会将相应的回调函数放入事件循环的队列中等待执行。所以，虽然V8引擎是Node.js的核心部分，用于执行JavaScript代码，但异步I/O操作是由libuv库和底层操作系统来处理的，而不是由V8引擎直接处理。这种设计使得Node.js能够高效地处理大量并发连接和异步操作，从而成为构建高性能网络应用的理想选择。

任务队列

在JavaScript中，宏任务（MacroTasks）和微任务（MicroTasks）是事件循环中处理异步操作的两类主要任务。

宏任务

  • script（全局任务）：这是代码的主入口，整个的script标签内都是宏任务。
  • setTimeout：延迟一段时间后执行的任务。
  • setInterval：每隔一段时间执行的任务。
  • setImmediate（Node.js环境）：在I/O事件后执行的任务，在Node.js中与setTimeout类似，但在浏览器中没有这个函数。
  • I/O操作：如Node.js中的文件读写、网络请求等。
  • UI渲染（浏览器环境）：浏览器在空闲时会进行页面的渲染。

微任务

  • Promise：Promise的回调函数是微任务。当Promise的状态改变（从pending变为fulfilled或rejected）时，其then或catch方法中的回调函数会被放入微任务队列。
  • process.nextTick（Node.js环境）：Node.js特有的API，用于在当前执行栈的末尾立即执行回调。但如果在Vue.js中，则为DOM更新之后立即执行。
  • MutationObserver（浏览器环境）：用于监视DOM更改的API，其回调函数也是微任务。

当JavaScript代码执行时，如果是在浏览器环境中，会先执行script（或称为全局任务）中的同步代码，然后执行微任务队列中的回调，最后才执行宏任务队列中的回调。这个过程会一直重复，直到所有的任务都被执行完毕。

事件循环

事件循环是 JavaScript 引擎处理异步任务和更新 UI 的主要方式。它的工作原理大致如下：

  1. 执行栈（Call Stack）：JavaScript 引擎有一个执行栈，用于存储函数调用和返回的结果。当执行栈为空时，事件循环会查看任务队列和微任务队列。

    • 同步代码：立即放入JS引擎（主线程）执行，并原地等待结果
    • 异步代码：先放入宿主环境（浏览器/Node)，不必原地等待结果，并不阻塞主线程继续往下执行，回调函数在等到异步结果之后再执行

    于是，调用栈中放主线任务，宿主环境来执行异步任务，在执行完后会通知主线程，并自动把回调函数推送到任务队列中，等执行栈中任务空了，就会从任务队列中拿回调。

    ![Untitled](Untitled 1.png)

    同样是异步任务的回调函数，有的回调是需要连贯执行的，为了优化这个体验，又分为了微任务（需要立即连贯执行）和宏任务（不需要立即连贯执行）

    事件循环中任务队列的过程可以分为先检查微任务队列，再检查宏任务队列

  2. 检查微任务队列：在每次事件循环迭代中，JavaScript 引擎会首先查看微任务队列。如果微任务队列中有任务，它会执行这些任务，直到微任务队列为空。

  3. 执行栈和微任务队列：在执行微任务的过程中，如果有新的微任务被添加到微任务队列，它们也会在当前事件循环迭代中被执行。只有当微任务队列完全为空时，JavaScript 引擎才会继续下一步。

  4. 检查宏任务队列：当微任务队列为空时，JavaScript 引擎会查看宏任务队列。如果宏任务队列中有任务，它会取出第一个任务并将其推入执行栈中执行。这个过程会一直重复，直到任务队列为空。

  5. 渲染：在执行完一个事件循环迭代中的所有任务后，浏览器会更新 UI 并进行渲染。

  6. 下一轮事件循环：然后，事件循环会开始新一轮的迭代，再次检查微任务队列和任务队列中的任务。

Promise

当一个请求，它依赖于上一个请求的结果的时候，就需要来调用回调函数，如果这样的请求多了，就会变成这样：

这就是回调地狱，为了优雅的表示异步回调，便有了Promise。

Promise 是 JavaScript 中的一个对象，它的构造函数为new Promise(fn)，fn为回调函数，它接收两个参数resolve和reject。

以下是 Promise 的一些基本概念和特性：

  1. 状态：Promise 有三种状态：

    • pending（待定）：初始状态，既不是成功，也不是失败状态。
    • fulfilled（已实现）：意味着操作成功完成。
    • rejected（已拒绝）：意味着操作失败。

  2. 执行器（Executor）：Promise 构造函数接受一个执行器函数作为参数，异步操作会放在该执行器函数内。执行器函数有两个参数：resolve 和 reject，它们是两个函数，分别用于将 Promise 的状态从 pending 变为 fulfilled 或 rejected。这两个函数为Promise对象的内置函数，无需自己定义。

    **resolve 函数被用来在异步操作成功时传递一个值给 Promise 的成功处理函数。这个值会成为 then 方法第一个回调函数的参数。**如果异步操作失败，你应该用 reject 函数，并传递一个表示错误原因的值给 Promise 的失败处理函数（即 then 方法的第二个参数，或者更常见的 catch 方法）。

  3. 链式调用（Chaining）：Promise 提供了 .then() 和 .catch() 方法，用于注册成功和失败的回调函数。这两个方法都返回新的 Promise 对象，因此可以链式调用。

  4. 错误处理：每一步都写reject特别麻烦，为了简化任务需要，我们诞生了catch方法，来捕获resolve时的异常，这样就可以省略reject方法了。

  5. Promise 链：多个 .then() 调用可以组成 Promise 链，每个 .then() 的回调函数都会接收前一个 Promise 的结果作为参数。

  6. 静态方法：为了满足更多业务需要，又诞生了6个promise的静态方法。

    • promise.all()等待所有Promise执行成功，一个不成功都算失败
    • promise.allSettled()等待所有Promise执行完成，无论成功还是失败
    • promise.any()任意一个promise成功就行
    • promise.race()等待第一个promise完成
    • promise.reject(reason)返回一个状态为rejected的Promise对象
    • Promise.resolve()它会接受四种类型的参数：
      1. 一个promise实例——直接返回
      2. 具有then方法的对象——转为promise对象，并立即执行then方法
      3. 普通对象或不是对象——返回一个新的promise对象，将参数传给下一个then
      4. 不带任何参数——状态为resolved的promise对象

async 和 await

async 和 await 是 JavaScript 中用于处理异步操作的关键字，它们主要用在 Promise 和基于 Promise 的技术中。使得异步代码能够以更直观、更易于理解的同步代码风格来编写。

async

  • async 函数是声明一个函数是异步的。
  • 异步函数总是返回一个 Promise。
  • 如果 async 函数中有 return 语句，返回的值将被包装在一个 Promise 中。
  • 如果 async 函数中没有 return 语句，则返回的 Promise 将解析为 undefined。

await

  • await 关键字只能在 async 函数内部使用。
  • await 会卡在当前的异步操作中等待 Promise 完成并返回结果，然后继续往下执行。
  • 如果等待的 Promise 被拒绝，await 表达式会抛出一个错误。

作用域

在JavaScript中，作用域（Scope）是代码执行上下文中的一个重要概念，它决定了变量、函数和表达式的可访问性和生命周期。作用域决定了在何处以及如何查找变量（标识符）的引用。

作用域类型

JavaScript中有两种主要的作用域类型：

  1. 全局作用域：不在任何函数内部的变量或函数，也就是定义在windows下的变量范围，在任何地方都可以访问。容易导致全局污染，命名冲突，对bug查找不利。
  2. 局部作用域：只在函数内部定义的变量范围。
    • 函数作用域：在ES6之前，JavaScript只有函数作用域，这意味着变量在函数内部是局部的，在函数外部是不可见的。
    • 块级作用域（Block Scope）：ES6引入了let和const关键字，它们提供了块级作用域。在{}中声明的变量或常量仅在该块内部是可访问的。

作用域链

作用域链就是由最内部的作用域往最外部查找变量的过程，形成的链条就是作用域链。

作用域链是一种数据结构，它保存了当前执行上下文中所有变量和函数引用的有序列表。这个列表从当前作用域开始，然后是其父级作用域，然后是更高级别的父级作用域，直到全局作用域。如果变量未在任何作用域中找到，那么将抛出一个引用错误（ReferenceError）。

闭包

闭包（Closure）允许函数记住并访问其所在的词法环境，即使在函数执行完毕并返回后，这些词法环境中的数据仍然可以被闭包内部的代码所访问。闭包有几个主要的应用场景：

  1. 数据封装和私有变量：

    闭包可以用来封装数据，并通过特定的接口（函数）来访问这些数据，从而实现数据的私有性。

  2. 回调函数和高阶函数：

    闭包经常作为回调函数使用，因为它们可以记住自己的词法环境，包括this、变量等。在高阶函数中，闭包也非常有用，因为它们可以接收函数作为参数，并返回函数作为结果。

  3. 实现装饰器/函数修饰器：

    闭包可以用来创建装饰器，它们可以修改或增强函数的行为。

  4. 实现模块模式：

    闭包可以用来模拟私有属性和方法，从而创建模块。这有助于避免全局命名空间的污染，并提供了一种组织代码的方式。

  5. 事件监听器：

    在事件驱动的编程中，闭包经常被用作事件监听器的回调函数，因为它们可以记住与特定事件相关的数据或状态。

原型和原型链

在 JavaScript 中，每个函数都有一个 prototype 属性，这个属性是一个指针，指向一个对象。这个对象的用途是包含可以由特定类型的所有实例共享的属性和方法，可以通过 Object.getPrototypeOf() 访问。

当你创建一个函数（即一个类构造函数）时，JavaScript 会自动为这个函数添加一个 prototype 属性，这个属性是一个对象，通常被称为该函数的原型对象。

在这个例子中，Person.prototype 就是 Person 构造函数的原型对象，而 sayHello 方法就定义在这个原型对象上。因此，当我们使用 new Person('Alice') 创建一个 Person 的实例时，这个实例（即 person1）可以访问到定义在 Person.prototype 上的 sayHello 方法。

原型链是 JavaScript 中实现继承的一种方式。当一个对象试图访问其某个属性或方法时，JavaScript 会首先在该对象自身的属性中查找，如果找不到，则会沿着该对象的原型链向上查找，直到找到为止（或者直到到达原型链的末尾，即 Object.prototype，此时如果还未找到，则返回 undefined）。

在上面的例子中，person1 对象并没有直接定义 sayHello 方法，但是它可以访问到 Person.prototype 上的 sayHello 方法，这就是因为 person1 的原型链中包含了 Person.prototype。

更具体地说，当我们使用 new 关键字创建一个新的对象时，JavaScript 会做以下几件事情：

  1. 创建一个新的空对象。
  2. 将这个新对象的 __proto__指向构造函数的 prototype 对象。
  3. 将这个新对象作为 this 上下文传入构造函数并执行，从而设置新对象的属性。
  4. 返回这个新对象。

因此，当我们说 person1.sayHello() 时，JavaScript 首先在 person1 对象自身上查找 sayHello 属性，由于没找到，所以会沿着原型链（即 person1.__proto__，也就是 Person.prototype）继续查找，最终在 Person.prototype 上找到了 sayHello 方法并执行。

需要注意的是，虽然 __proto__ 属性在大多数现代浏览器中都可以访问，但它并不是 JavaScript 语言规范的一部分，而是某些浏览器为了实现原型链而提供的。在编写可移植的代码时，应该使用符合JavaScript 语言规范的 Object.getPrototypeOf() 和 Object.setPrototypeOf() 方法来访问和修改对象的原型。

什么是模块化？

概括：模块化的目的是当系统的规模和复杂度上升后，将代码拆分成多个独立、可重用的单元，提升代码的复用性，避免全局变量污染

阐述：将一些共有或者私有的变量和方法封装为一个模块后导出，来让别的模块引用。每个模块都具有特定的功能和职责，并且可以与其他模块进行交互。

总结：一个模块的基本内容是：依赖的模块、自己要做的事情、要导出的变量/方法

es6和commonJS区别

ES6 module 静态编译时加载，输出的是接口，CommonJS 动态运行时加载，加载的是一个对象

ES6 module支持异步加载，浏览器中会用到该特性，而CommonJS是不支持异步的，因为服务器端不需要异步加载。所以CommonJS不可替代ES6 module，ES6 module可以替代CommonJS。

  1. 语法：

    • ES6 使用 import 和 export 关键字来导入和导出模块，更简洁直观。
    • CommonJS 使用 require() 函数来导入模块，并使用 module.exports 或 exports 对象来导出模块。

  2. 静态与动态：

    • ES6 模块是静态的，这意味着模块的依赖关系在编译时就确定了，模块会在文件执行之前被加载和执行。但 ES6 也支持异步加载（如动态 import()）。这意味着模块的加载不会阻塞后续代码的执行，而是会返回一个Promise，该Promise在模块加载完成后才会被解析，因此，它也可以实现类似 CommonJS 的实现方式。

    • CommonJS 模块是动态的，它在运行时解析模块的依赖关系，因此可能会导致一些运行时的开销。当一个模块被引入时，它会立即执行并且等待加载完成，然后才继续执行后续代码。比如，以下写法在 CommonJS 中是支持的：

  3. 缓存：

    • ES6 模块的导入是单例模式，也就是说，一个模块只会被加载和执行一次，后续的导入都会返回同一个实例。
    • CommonJS 模块的导入是值拷贝，每次导入都会创建一个新的实例。

  4. 浏览器和Node.js兼容性：

    • ES6 模块的兼容性在浏览器中逐渐提升，但在Node.js中还需要使用 -experimental-modules 标志启用（虽然现在Node.js已经支持ES6模块作为稳定特性）。
    • CommonJS 主要在Node.js环境中使用，并已被广泛接受和支持。

  5. 其他：

    • CommonJS 中顶层的 this 指向这个模块本身，而 ES6 中顶层 this 指向 undefined
    • CommonJS 是可以修改模块值的，esm值并且不可修改（可读的）
    • CommonJS 不可以tree shaking，esm支持tree shaking，因为在esm在编译阶段已经知道需要知道那些模块了。

Package json

执行npm init便可以初始化一个package.json，括号内为默认选项，加上-y参数则将所有选项填充为默认值。

  1. name：项目名称，必须是唯一的字符串，通常采用小写字母和连字符的组合。

  2. version：项目版本号，通常采用语义化版本号规范。

    version 三段式版本号（例如1.0.0 ）分别代表大版本号、次版本号 、修订号，
    大版本号一般是有重大变化才会升级，
    次版本号一般是增加功能进行升级，
    修订号一般是修改bug进行升级

  3. description：项目描述。

  4. main：项目的主入口文件路径，通常是一个 JavaScript 文件。

  5. keywords：项目的关键字列表，方便他人搜索和发现该项目。

  6. author：项目作者的信息，包括姓名、邮箱、网址等。

  7. license：项目的许可证类型，可以是自定义的许可证类型或者常见的开源许可证（如 MIT、Apache 等）。

  8. dependencies：项目生产环境所依赖的包的列表，这些包会在项目运行时自动安装。如果想在不同环境下配置不同的依赖项，可以使用以下两个配置：

    • devDependencies：仅在开发过程中所需要的包的列表，这些包不会随项目一起发布，而是只在开发时使用。

    • peerDependencies：项目的同级依赖，即项目所需要的模块被其他模块所依赖。一般是给编写插件和npm包的人员所使用的，比如插件名为vite-plugin-xm，插件的依赖为vite ：

  9. scripts：定义了一些脚本命令，比如启动项目、运行测试等。这些命令是通过npm run xxx 来执行的。

  10. repository：项目代码仓库的信息，包括类型、网址等。

  11. bugs：项目的 bug 报告地址。

  12. homepage：项目的官方网站地址或者文档地址，将npm包发布到npm官网上之后，以上配置都可以被看到。

npm 命令

  1. npm init：初始化一个新的 npm 项目，生成 package.json 文件。

  2. npm install：安装一个包或一组包（默认为最新的，可以通过@来下载指定版本），简写为npm i。安装的依赖都会存放在根目录的node_modules，默认采用扁平化的方式安装，并且排序规则为.>@>首字母。使用的算法是广度优先遍历，在遍历依赖树时，npm会首先处理项目根目录下的依赖，然后逐层处理每个依赖包的依赖，直到所有依赖都被处理完毕。在处理每个依赖时，npm会检查该依赖的版本号是否符合依赖树中其他依赖的版本要求，如果不符合，则会尝试安装适合的版本。但是这种扁平化只存在于理想情况下，如果有多个模块用到同一个依赖，会出现复用的情况：

    • 安装某个二级模块时，若发现第一层级有相同名称，相同版本的模块，便直接复用那个模块
    • 安装某个二级模块时，若发现第一层级有相同名称，但不同版本的模块，会下载多个版本的该模块，还是会出现模块冗余的情况

    比如A 依赖 C1.0，B 依赖 C1.0，D 依赖 C2.0。此时C 1.0就会被放到A B的node_moduels，C2.0 会被放入D模块下面的node_moduels。

  3. npm install <package-name> --save：安装指定的包，并将其添加到 package.json 文件中的dependencies依赖列表中，在npm5.4版本之后加不加没区别。

  4. npm install <package-name> --save-dev：安装指定的包，并将其添加到 package.json 文件中的开发依赖列表中，简写为npm -D。

  5. npm install -g <package-name>：全局安装指定的包。

  6. npm update <package-name>：更新指定的包。

  7. npm uninstall <package-name>：卸载指定的包。

  8. npm run <script-name>：执行 package.json 文件中定义的脚本命令。

  9. npm search <keyword>：搜索 npm 库中包含指定关键字的包。

  10. npm info <package-name>：查看指定包的详细信息。

  11. npm list：列出当前项目中安装的所有包。

  12. npm outdated：列出当前项目中需要更新的包。

  13. npm audit：检查当前项目中的依赖项是否存在安全漏洞。

  14. npm publish：发布自己开发的包到 npm 库中。

  15. npm login：登录 npm 账户。

  16. npm logout：注销当前 npm 账户。

  17. npm link: 将本地模块链接到全局的 node_modules 目录下

  18. **npm config list** 用于列出所有的 npm 配置信息。执行该命令可以查看当前系统和用户级别的所有 npm 配置信息，以及当前项目的配置信息（如果在项目目录下执行该命令）

  19. npm get registry 用于获取当前 npm 配置中的 registry 配置项的值。registry 配置项用于指定 npm 包的下载地址，如果未指定，则默认使用 npm 官方的包注册表地址

  20. npm config set registry <registry-url> 命令，将 registry 配置项的值修改为指定的 <registry-url> 地址，比如淘宝镜像。

npm install 后续流程

首先，检查config里的文件（可以通过npm config —list来查看），首先检查项目级别的.npmrc，至于npmrc可以配置什么给大家一个demo参考。如果没有进行配置过的话，会按照用户级别>全局级别>node内置级别的顺序去找.npmrc配置文件。

找到配置后会检查是否存在package-lock.json文件：

  • 如果存在会比较package-lock.json和package.json的版本是否一致：
    • 如果不一致，会根据package.json的版本进行下载，并对package-lock.json文件进行更新。
    • 如果一致，会检查是否存在需要安装模块的缓存，也就是之前安装过该模块，会直接把该模块解压到node_modules中，如果不存在的话，则去npm官网下载该资源包，并检查其完整性，添加到缓存中，更新lock文件，最后解压到node_modules中
  • 如果不存在，获取包的信息，扁平化构建依赖树，再来检查有没有缓存，后续过程和以上内容没差别

package-lock.json 的作用

很多朋友只知道这个东西可以锁定版本记录依赖树详细信息，它的一些具体参数含义为：

  • version 参数指定了当前包的版本号
  • resolved 参数指定了当前包的下载地址
  • integrity 是一串用于验证包的完整性的哈希值
  • dev 参数指定了当前包是一个开发依赖包
  • bin 参数指定了当前包中可执行文件的路径和名称
  • engines 参数指定了当前包所依赖的Node.js版本范围

知识点来了，package-lock.json 帮我们做了缓存，他会通过 name + version + integrity 信息生成一个唯一的key，这个key能找到对应的index-v5 下的缓存记录，如果发现有缓存记录，就会找到tar包的hash值，然后将对应的二进制文件解压到node_modeules

NPX

npx是一个命令行工具，它是npm 5.2.0版本中新增的功能。它允许用户在不安装全局包的情况下，运行已安装在本地项目中的包或者远程仓库中的包的可执行文件。具有以下优点：

  1. 避免全局污染：npx允许你执行npm package，而不需要你先全局安装它
  2. 总是使用最新版本：如果你没有在本地安装相应的npm package，npx会从npm的package仓库中下载并使用最新版，执行完之后再把包删掉
  3. 执行任意npm包：npx不仅可以执行在package.json的scripts部分定义的命令，还可以执行任何npm package，甚至可以执行GitHub gist或者其他公开的JavaScript文件
💡 `*npm ls -g` 查看全局安装的可执行文件，不加`-g`就是查看当前项目*

npm 和 npx 区别：

  1. npx 侧重于执行命令的，执行某个模块命令。虽然会自动安装模块，但是重在执行某个命令。它有两种使用场景：

    • 创建一个react项目，npm install -g create-react-app ，将该模块安装到全局之后才能执行 create-react-app my-app，这样的话会有两个问题：

      1. 首先需要全局安装这个包占用磁盘空间
      2. 如果模块需要更新还得执行更新命令

      因此，使用npx不仅不占用磁盘空间，而且能保证执行的命令为最新版，但是每次执行都要等待它重新下载

    • 在本地项目中执行vite，但是我全局并没有安装vite，仅在当前项目npm i vite -D安装了vite，由于当前项目没有配置环境变量，因此是无法直接执行该命令的，使用npx vite 可以直接运行未全局安装的npm包，因为npx 的运行规则和npm 是一样的，本地目录查找.bin ，如果没有就去全局的node_moduels 查找，如果还没有就去下载这个包然后运行命令，然后删除这个包

  2. npm侧重于安装或者卸载某个模块，重在安装，并不具备执行某个模块的功能。

NPM搭建私服

我们平时发布npm包是发在npm官方的镜像源上，npm的私服则是个人仓库，基本上每个公司都会有自己的npm私服，发布一些内部包。构建npm私服的优点：

  1. 可以离线使用，你可以将npm私服部署到内网集群，这样离线也可以访问私有的包。
  2. 提高包的安全性，使用私有的npm仓库可以更好的管理你的包，避免在使用公共的npm包的时候出现漏洞。比如公共npm包如果出现了问题，私有仓库可以选择不进行同步，维持原来版本。
  3. 提高包的下载速度，使用私有 npm 仓库，你可以将经常使用的 npm 包缓存到本地，从而显著提高包的下载速度，减少依赖包的下载时间。这对于团队内部开发和持续集成、部署等场景非常有用。

Node.js介绍

介绍

概述

  1. NodeJs不是JavaScript应用，也不是编程语言，因为编程语言用的是JavaScript，Nodejs是 JavaScript的运行时的环境。

  2. Nodejs是构建在V8引擎之上的，V8引擎是由C/C++编写的，因此我们的JavaSCript代码需要由C/C++转化后再执行。

  3. NodeJs 使用异步 I/O 和事件驱动的设计理念，可以高效地处理大量并发请求，提供了非阻塞式 I/O 接口和事件循环机制，使得开发人员可以编写高性能、可扩展的应用程序，异步I/O最终都是由libuv 事件循环库去实现的。

  4. NodeJs 使用npm作为包管理工具，类似于python的pip，或者是java的Maven，目前npm拥有上百万个模块。

  5. NodeJs适合干一些IO密集型应用，不适合CPU密集型应用，nodejsIO依靠libuv有很强的处理能力，而CPU因为NodeJs单线程原因，容易造成CPU占用率高，如果非要做CPU密集型应用，可以使用C++插件编写，或者NodeJs提供的cluster。(CPU密集型指的是图像的处理或者音频处理需要大量数据结构 + 算法的运算过程)

    IO意为Input和Output，也就是输入输出。

NodeJs 大致架构图

最上层为应用层，也就是可供开发的API，由JavaScript编写。

中间层和底层实现通讯，由C和C++来编写。

同时，NodeJs在做一些不擅长的事情的时候，可以使用C或C++写的插件来做。

底层需要使用V8引擎来解析JavaScript，最底层为操作系统，NodeJs可以实现跨平台。

Nodejs与浏览器环境

运行在浏览器环境中的代码：

  • HTML页面中的JavaScript：在HTML文件的<script>标签中直接嵌入的JavaScript代码。这些代码通常用于处理用户的交互事件（如点击按钮）、更新DOM（如添加或修改页面元素）、发送AJAX请求等。
  • 外部JavaScript文件：通过<script src="...">标签在HTML页面中引入的外部JavaScript文件。这些文件通常包含更复杂的逻辑和函数，用于处理页面上的各种功能和交互。

script.js 文件的内容：

  • 浏览器特定的API：使用浏览器提供的特定API，如window、document、navigator、localStorage、fetch API等。

运行在Node.js环境中的代码：

  • 命令行工具：使用Node.js编写的命令行工具，可以执行各种任务，如文件操作、网络请求、数据转换等。
  • Web服务器：使用Node.js的HTTP或Express等框架创建的Web服务器，用于处理客户端的请求并返回响应。
  • 后台任务：使用Node.js编写的长时间运行的后台任务，如数据抓取、图像处理、日志分析等。
  • 数据库操作：使用Node.js连接和操作数据库，如MySQL、MongoDB等。
  • Node.js特定的模块和API：使用Node.js提供的核心模块（如fs、path、http）或第三方模块（如Express、Koa、Mongoose）来编写代码。
  • 没有DOM操作：Node.js代码通常不包含直接操作DOM的代码，因为它在服务器端运行，没有DOM环境。

总结来说，运行在浏览器环境中的代码主要用于处理用户交互、更新页面内容、发送网络请求等前端任务，而运行在Node.js环境中的代码则主要用于构建服务器端应用、处理数据、创建命令行工具等后端任务。两者在使用的API、库、工具和运行环境上都有所不同。

运行在浏览器环境和Node.js环境的代码主要存在以下区别：

  1. 全局对象：
    • 在浏览器环境中，全局对象是window，它提供了与浏览器窗口和文档交互的方法和属性。
    • 在Node.js环境中，全局对象是global，但它并不提供与DOM或浏览器窗口相关的功能。
  2. 运行环境：
    • 浏览器环境主要用于运行前端代码，与用户直接交互，如操作DOM、处理用户事件等。
    • Node.js环境主要用于运行服务器端代码，如文件操作、数据库操作、网络请求等。Node.js提供了许多API，如fs、http、crypto、path等，用于处理这些服务器端任务。
  3. 模块系统：
    • 浏览器环境原生支持ES6模块，并可以通过工具如webpack支持CommonJS或AMD等模块系统。
    • Node.js环境原生支持CommonJS模块，并已经开始支持ES6模块。在Node.js中，模块是通过require()函数引入的，而在浏览器中，可以使用ES6的import语句或其他模块加载器。
  4. DOM操作：
    • 浏览器环境提供了对DOM（文档对象模型）的完整访问，允许JavaScript代码直接操作HTML和CSS。
    • Node.js环境没有DOM，因此不能直接操作HTML或CSS。然而，Node.js可以通过一些库（如jsdom）来模拟DOM环境，但这些库主要用于测试或其他特殊用途。
  5. 安全性：
    • 浏览器环境的安全性受到同源策略（Same-Origin Policy）的限制，以防止跨站脚本攻击（XSS）和其他安全问题。浏览器中的JavaScript代码不能直接访问其他来源的数据或资源。
    • Node.js环境在服务器端运行，因此具有更高的权限和更少的限制。然而，这也增加了潜在的安全风险，因此开发者需要格外小心，避免引入安全漏洞。
  6. I/O操作：
    • 在浏览器环境中，JavaScript代码通常通过AJAX或Fetch API等技术与服务器进行异步通信。这些操作是异步的，不会阻塞主线程。
    • 在Node.js环境中，JavaScript代码可以直接与文件系统、数据库和其他I/O设备进行交互。Node.js使用事件驱动和非阻塞I/O模型来处理这些操作，以实现高并发和性能。
  7. 扩展性：
    • 浏览器环境主要关注前端功能的实现和用户体验的优化，因此其API和功能相对有限。
    • Node.js环境则更加灵活和可扩展，它可以通过npm（Node Package Manager）安装和使用大量的第三方库和工具来扩展其功能和应用范围。

安装NodeJs

  • 如果你是一个初学者或只需要在单一项目中工作：直接安装 Node.js
    1. 当安装 Node.js 时，npm 通常会自动作为一部分一起安装，因为 npm 是 Node.js 的默认包管理工具。它允许开发者从 npm 仓库中安装、更新、发布和管理 Node.js 的第三方包和依赖。
    2. npm 还支持使用 package.json 文件来管理项目的依赖和元数据，因此在项目中使用npm install即可将依赖包下载完成，
  • 如果你需要维护多个项目，并且这些项目需要不同版本的 Node.js：安装 nvm 可以更好地管理不同版本的 Node.js，避免版本冲突。
    1. 使用 nvm 安装或卸载特定版本Node.js：nvm install 16.16.0 和nvm uninstall 16.16.0
    2. 切换 Node.js 版本：nvm use 14.15.3
    3. 查看已安装和当前正在使用的 Node.js 版本：nvm list

安装成功后可以输入node -v && npm -v && npx -v 来查看是否安装成功及安装版本

Git是目前世界上最先进的分布式版本控制系统：

  • 版本控制系统： 记录文件的变化，以便进行追踪和管理
  • 分布式系统： 意味着包括版本历史的整个代码仓库不仅存储在中央服务器上，而且每个开发者的本地机器上也有完整的复制

本地 Git 操作

初始化仓库

当使用 git init 命令时，Git 会在当前目录下创建一个新的 .git 目录。这个目录是 Git 仓库的核心，包含了所有的版本控制信息，如提交历史、分支、标签等。并且默认进入 Git 仓库的master分支，即主分支。该分支为空，需要对文件手动进行追踪。

跟踪文件

Git 会跟踪你添加到仓库中的文件，你可以使用 git add 命令将文件添加到暂存区，然后使用 git commit 命令将暂存区的更改提交到仓库中。

  1. 跟踪与忽略：Git 默认会跟踪添加到暂存区的所有文件。但是，你可以通过 .gitignore 文件来指定 Git 应该忽略哪些文件或目录。这通常用于排除自动生成的文件、临时文件或敏感信息。

  2. 未跟踪文件：对于你未使用 git add 添加的文件，Git 会将它们视为未跟踪文件。这些文件不会出现在 Git 的版本控制中，除非你显式地将它们添加到暂存区。

  3. 已跟踪文件的更改：对于已经添加到 Git 仓库并被跟踪的文件，任何对它们的更改都会被 Git 记录下来。

记录更改或版本回退

每次提交时，Git 都会记录你对文件的更改。这些更改以快照的形式保存，包含了文件在提交时的完整状态。

Git 提供了强大的历史查看功能，你可以使用 git log 命令查看提交历史，包括每次提交的哈希值、作者、日期、消息和更改的文件，通过历史记录可以进行版本回退。

  • 硬回退

    如果你想要完全回退到某个提交，并且丢弃该提交之后的所有更改，你可以使用 git reset 命令。这通常被称为“硬回退”，因为它会改变你的工作目录和暂存区以匹配目标提交，并且改变历史记录。

    💡 警告：这个操作会丢失目标提交之后的所有更改，所以请确保你真的想要这么做。

远程仓库

首先，你必须要有一个远程仓库，一般在 GitHub、GitLab 或其他 Git 托管服务上，以Github为例，需要申请一个账户，如果以后我们要在 GitHub 上搭建自己的个人博客，其默认地址就是username.github.io。

GitHub 术语解释

Repository：简称Repo，可以理解为“仓库”，一个仓库存放一个项目。也就是说，如果我们想要建立项目，就得先建立仓库；有多个项目，就建立多个仓库。

Issues：可以理解为“问题”，举一个简单的例子，如果我们开源一个项目，如果别人看了我们的项目，并且发现了bug，或者感觉那个地方有待改进，他就可以给我们提出Issue，等我们把Issues解决之后，就可以把这些Issues关闭；反之，我们也可以给他人提出Issue。

Star：可以理解为“点赞”，当我们感觉某一个项目做的比较好之后，就可以为这个项目点赞，而且我们点赞过的项目，都会保存到我们的Star之中，方便我们随时查看。

Fork：可以理解为“拉分支”，如果我们对某一个项目比较感兴趣，并且想在此基础之上开发新的功能，这时我们就可以Fork这个项目，这表示复制一个完全相同的项目到我们的 GitHub 账号之中，而且独立于原项目。之后，我们就可以在自己复制的项目中进行开发了。

Pull Request：可以理解为“提交请求”，此功能是建立在Fork之上的，如果我们Fork了一个项目，对其进行了修改，而且感觉修改的还不错，我们就可以对原项目的拥有者提出一个Pull请求，等其对我们的请求审核，并且通过审核之后，就可以把我们修改过的内容合并到原项目之中，这时我们就成了该项目的贡献者。

Merge：可以理解为“合并”，如果别人Fork了我们的项目，对其进行了修改，并且提出了Pull请求，这时我们就可以对这个Pull请求进行审核。如果这个Pull请求的内容满足我们的要求，并且跟我们原有的项目没有冲突的话，就可以将其合并到我们的项目之中。当然，是否进行合并，由我们决定。

Watch：可以理解为“观察”，如果我们Watch了一个项目，之后，如果这个项目有了任何更新，我们都会在第一时候收到该项目的更新通知。

Gist：Github中的Gist是一个子服务，它允许用户在线分享和存储较小的代码片段、笔记、错误信息等内容。例如，你正在解决一个具体的编程问题，并找到了一个简洁的代码片段来解决这个问题。你可以将这个代码片段复制并粘贴到Gist中，然后通过链接的方式分享给其他人。其他人可以直接点击链接查看你的代码片段，而无需下载整个项目或仓库。

利用 SSH 完成 Git 与 GitHub 的绑定

第 1 步：生成 SSH key

我们要想生成SSH key，首先就得先安装 SSH，对于 Linux 和 Mac 系统，其默认是安装 SSH 的，而对于 Windows 系统，其默认是不安装 SSH 的，不过由于我们安装了 Git Bash，应该也自带了 SSH。可以通过在 Git Bash 中输入ssh命令，查看本机是否安装 SSH。

接下来，输入ssh-keygen -t rsa命令，表示我们指定 RSA 算法生成密钥，之后就就会生成两个文件，分别为密钥id_rsa和公钥id_rsa.pub。

第 2 步：添加 SSH key

密钥和公钥生成之后，需要把公钥id_rsa.pub的内容添加到 GitHub的SSH keys中，这样我们本地的密钥id_rsa和 GitHub 上的公钥id_rsa.pub才可以进行匹配，授权成功后，就可以向 GitHub 提交代码啦。

第 3 步：验证绑定是否成功

在我们添加完SSH key之后，可以通过在 Git Bash 中输入ssh -T git@github.com进行测试，如果显示授权成功，则表示Git 与 GitHub 已经绑定，这意味着我们已经可以通过 Git 向 GitHub 提交代码啦！

分支操作

在进行任何的git操作之前，都得先切换到 Git 的仓库最顶层目录，然后从此目录中进入 Git Bash，这样之后的操作才能顺利进行。

  1. 查看分支：
    • git branch：查看当前仓库中的所有分支，以及当前所在的分支（显示为带有星号*的分支）。
    • git branch -r：查看远程分支。
    • git branch -a：查看本地和远程分支。
  2. 创建分支：
    • git branch <branch-name>：创建一个新的分支，将<branch-name>替换为您要创建的分支的名称。
  3. 切换分支：
    • git checkout <branch-name>：切换到新创建的分支，将<branch-name>替换为要切换的分支的名称。
  4. 删除分支：
    • git branch -d <branch-name>：删除本地分支，如果分支上有未合并的修改，Git会提示你进行确认。
    • git branch -D <branch-name>：强制删除本地分支。
    • git push <remote_name> --delete <branch_name>：删除远程分支，其中<remote_name>是你的远程仓库的名称，<branch_name>是你要删除的分支的名称。
  5. 合并分支：
    • git merge <branch-name>：将指定分支合并到当前分支。例如，如果你在master分支上，想要合并dev分支，可以使用git merge dev。
    • 如果只想合并指定分支上的某个提交，可以使用git cherry-pick <commit-hash>。

初始化远程仓库

  1. 添加远程仓库：将文件添加到暂存区并提交暂存区中的更改后，想要推送到远程仓库，需要使用 git remote add 命令将Github远程仓库添加进来。

    通常，origin 是远程仓库的默认名称。

  2. 推送本地分支到远程仓库：

    -u 或 --set-upstream 选项用于设置本地分支跟踪远程分支，这样在以后的推送和拉取操作中，Git 会知道对应的远程分支是什么。

  3. 验证推送：

    推送完成后，登录到你的远程仓库（GitHub 或 GitLab），检查是否已经看到了你推送的代码。

合作开发

加入一个项目后，需要先通过git clone命令克隆项目的远程仓库到本地，接下来通常需要执行以下步骤：

  1. 运行npm install来安装依赖

  2. 配置开发环境：根据项目需求，配置你的开发环境，比如设置环境变量、安装数据库、配置API密钥等。

  3. 了解项目的分支策略：git branch 查看项目分支，通常会基于某个特定的分支（如dev或feature）创建一个新的功能分支来进行开发工作。

  4. 创建并切换到新的分支：为了开发新功能或修复bug，你应该创建一个新的分支。这样做可以保持主分支的干净，并且允许你的工作在被合并之前进行审查和测试。

  5. 进行开发工作

  6. 提交更改：当你想要推送本地提交到远程仓库之前，需要确保你的本地仓库与远程仓库同步，以避免潜在的合并冲突。以下是推荐的步骤：

    第 1 步：更新本地仓库

    获取远程仓库的最新更改，防止覆盖别人的代码或者产生冲突。

    第 2 步：解决冲突

    在当前目录中查看拉取到的代码，检查是否有冲突，如果存在冲突，解决之后需要使用git add和git commit重新提交。

    第 3步：推送更改到远程仓库

  7. 创建合并请求（Pull Request）：

    在你的代码推送到远程仓库后，你通常需要创建一个合并请求（Pull Request，简称PR），以便项目维护者可以审查你的代码并决定是否将其合并到主分支。

  8. 参与代码审查：

    等待其他开发者的反馈，并根据需要进行修改。这是一个协作过程，所以请保持开放和积极的态度对待他人的建议和评论。

  9. 等待合并：

    一旦你的PR通过了审查并且所有问题都得到了解决，项目维护者会将你的更改合并到主分支中。此时，你的贡献就已经成为了项目的一部分！
